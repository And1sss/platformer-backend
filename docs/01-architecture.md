## 1.1. Задание к разделу «Архитектура»

### 1.1.1. Цель
Целью разработки архитектуры является проектирование информационной системы сопровождения игрового продукта, которая обеспечивает взаимодействие клиентского приложения на Unity с серверной частью, выполняющей авторизацию пользователей, обработку игровых событий, хранение прогресса, подсчёт статистики и формирование таблицы лидеров.

1.2.2. Пояснение к архитектуре

Клиентом в проекте выступает 2D-платформер, разработанный в Unity. Клиентское приложение отвечает за пользовательский интерфейс, игровой процесс, отправку запросов на сервер и получение ответов от серверной части.

API реализуется в виде REST-интерфейса на Flask и обеспечивает обмен данными между клиентом и сервером в формате JSON по протоколу HTTPS.

Модуль Auth выполняет проверку JWT-токена, извлекает идентификатор пользователя и его роли, а также ограничивает доступ к защищённым endpoint'ам в соответствии с моделью RBAC.

Слой Business Logic реализует основную серверную логику проекта. Он отвечает за обработку попыток прохождения уровней, фиксацию игровых событий, расчёт наград, обновление прогресса пользователя, статистики и лидерборда.

Data Layer представлен базой данных MySQL 8.0, в которой хранятся аккаунты пользователей, роли, игровые сессии, игровые события, прогресс, статистика и результаты прохождения уровней. Дополнительно могут использоваться Redis как кэш и Message Queue для асинхронной обработки событий, однако в минимальной версии системы эти компоненты являются необязательными.

1.3.1. Выбранный endpoint

POST /api/v1/level/finish

Данный endpoint используется для завершения попытки прохождения уровня. Сервер проверяет пользователя, валидирует входные данные, фиксирует результат прохождения, рассчитывает награды и обновляет связанные данные в базе.

POST /api/v1/level/finish
Authorization: Bearer <JWT>
Content-Type: application/json

{
  "attemptId": 15,
  "result": {
    "isCompleted": true,
    "score": 1850,
    "durationSeconds": 142,
    "coinsCollected": 18,
    "enemiesDefeated": 6
  }
}

1.5. Auth (JWT + RBAC)

При обращении к endpoint POST /api/v1/level/finish сервер сначала проверяет наличие и корректность JWT-токена. Из токена извлекается идентификатор пользователя и список его ролей. Доступ к endpoint разрешается пользователю с ролью player.

Если токен отсутствует, повреждён, просрочен или роль пользователя не подходит, сервер возвращает ошибку авторизации.

Возможная ошибка:

{
  "ok": false,
  "error": {
    "code": "UNAUTHORIZED",
    "message": "JWT is missing/invalid/expired",
    "requestId": "req_abc123"
  }
}

1.6. Validation

После прохождения авторизации сервер валидирует тело запроса. Проверяется, что:

attemptId является целым числом
объект result присутствует в запросе
result.isCompleted имеет тип boolean
result.score является целым числом и не меньше 0
result.durationSeconds является целым числом и не меньше 0
result.coinsCollected является целым числом и не меньше 0
result.enemiesDefeated является целым числом и не меньше 0

Дополнительно сервер проверяет:

существует ли попытка прохождения уровня
принадлежит ли попытка текущему пользователю
не была ли попытка уже завершена

Возможная ошибка:

{
  "ok": false,
  "error": {
    "code": "CONFLICT",
    "message": "Level attempt is already finished or does not belong to the user",
    "requestId": "req_abc123"
  }
}

1.7. Business Logic

Серверная логика выполняет следующие действия:

Проверяет, что попытка прохождения уровня существует и имеет статус started.
Подтверждает, что попытка принадлежит пользователю, идентификатор которого извлечён из JWT.
Рассчитывает награды на сервере, не доверяя клиенту.
Завершает попытку прохождения уровня.
Сохраняет итоговый результат.
Обновляет прогресс игрока.
Обновляет статистику за текущий день.
Обновляет лидерборд.

Клиент не является источником истины. Клиент отправляет только факты прохождения уровня, а итоговые награды и обновления вычисляются сервером.

Пример расчёта наград:

если уровень завершён успешно:
xp_gained = 100
soft_currency_gained = 50
если уровень не завершён:
xp_gained = 25
soft_currency_gained = 10

1.8. Data Layer (MySQL, транзакция)
1.8.1. Начало транзакции
START TRANSACTION;

1.8.2. SQL-запросы
-- 1. Проверка попытки прохождения уровня
SELECT id, user_id, status
FROM level_attempts
WHERE id = 15;

-- 2. Завершение попытки
UPDATE level_attempts
SET status = 'finished',
    ended_at = NOW(3)
WHERE id = 15;

-- 3. Запись результата
INSERT INTO level_results
(attempt_id, is_completed, score, duration_seconds, coins_collected, enemies_defeated)
VALUES
(15, 1, 1850, 142, 18, 6);

-- 4. Блокировка строки прогресса пользователя
SELECT user_id
FROM player_progress
WHERE user_id = 1
FOR UPDATE;

-- 5. Обновление прогресса
UPDATE player_progress
SET xp = xp + 100,
    soft_currency = soft_currency + 50
WHERE user_id = 1;

-- 6. Обновление статистики
INSERT INTO statistics_daily
(user_id, day, sessions_count, events_count, playtime_seconds, wins, losses, score_sum)
VALUES
(1, CURDATE(), 0, 1, 142, 1, 0, 1850)
ON DUPLICATE KEY UPDATE
events_count = events_count + 1,
playtime_seconds = playtime_seconds + VALUES(playtime_seconds),
wins = wins + VALUES(wins),
losses = losses + VALUES(losses),
score_sum = score_sum + VALUES(score_sum);

-- 7. Обновление лидерборда
INSERT INTO leaderboard_scores (user_id, board_code, season, score)
VALUES (1, 'platformer_score', 1, 1850)
ON DUPLICATE KEY UPDATE
score = GREATEST(score, VALUES(score)),
updated_at = CURRENT_TIMESTAMP(3);

1.8.3. Завершение транзакции
COMMIT;

1.10. Возможные ошибки
1.10.1. 401 Unauthorized

Причина: JWT отсутствует, повреждён, просрочен или не может быть проверен сервером.

1.10.2. 409 Conflict

Причина: попытка прохождения уровня уже была завершена либо не принадлежит текущему пользователю.

1.10.3. 404 Not Found

Причина: attemptId не существует в базе данных.

1.11. Архитектурные принципы проекта
Сервер является источником истины.
Все критически важные изменения выполняются в транзакции.
API реализуется по принципу Stateless REST.
Архитектура разделена на слои: routes, controllers, services, repositories, DB.
Для защиты от race condition используется блокировка SELECT ... FOR UPDATE.
Для защиты от повторной отправки событий может использоваться дедупликация через processed_events.

1.12. Адаптация под жанр

Для жанра 2D-платформер в архитектуре важно учитывать следующие особенности:

прохождение уровня рассматривается как отдельная попытка level_attempt
сервер фиксирует факт завершения уровня и его результат
клиент не рассчитывает награды самостоятельно
статистика строится по времени прохождения, количеству завершённых уровней, поражений и очков
лидерборд формируется по наибольшему количеству очков или лучшему времени прохождения

1.13. Итог

Разработанная архитектура соответствует модели Client → API → Auth → Business Logic → DB. В системе предусмотрены серверная валидация, контроль доступа, транзакционная обработка результатов, хранение прогресса и статистики в базе данных, а также возможность масштабирования за счёт разделения логики по слоям и использования стандартного REST-подхода.
