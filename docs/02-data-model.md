## 2.1. Задание к разделу 2 «Универсальная модель данных»

### 2.1.1. Цель
Целью данного раздела является проектирование универсального ядра базы данных информационной системы сопровождения игрового продукта, пригодного для хранения пользователей, ролей, игровых сессий, событий, прогресса, статистики и таблиц лидеров.

---

## 2.2. Сущности ядра

### users
Таблица `users` предназначена для хранения аккаунтов пользователей системы. В ней сохраняются уникальный email, bcrypt-хэш пароля, никнейм, дата создания аккаунта, дата последнего входа и признак блокировки пользователя.

### roles
Таблица `roles` хранит справочник ролей доступа. В проекте могут использоваться роли `player`, `moderator` и `admin`.

### user_roles
Таблица `user_roles` реализует связь многие-ко-многим между пользователями и ролями. Она позволяет назначать одному пользователю одну или несколько ролей.

### game_sessions
Таблица `game_sessions` хранит игровые сессии пользователя. В ней фиксируется платформа клиента, версия клиента, IP-адрес, время начала и завершения игровой сессии.

### game_events
Таблица `game_events` используется для хранения игровых событий. Она содержит тип события, JSON-представление полезной нагрузки, ссылку на пользователя и, при необходимости, ссылку на игровую сессию.

### player_progress
Таблица `player_progress` хранит текущее агрегированное состояние игрока. В ней содержатся уровень, количество опыта, мягкая и твёрдая валюта, а также дата последнего обновления.

### statistics_daily
Таблица `statistics_daily` хранит ежедневную статистику пользователя. В ней фиксируются количество сессий, количество событий, общее время в игре, победы, поражения и суммарное количество очков за день.

### leaderboard_scores
Таблица `leaderboard_scores` хранит данные таблиц лидеров. Она содержит код таблицы, сезон, очки пользователя и дату обновления результата.

## 2.4. Назначение ядра модели данных

Универсальное ядро модели данных необходимо для того, чтобы серверная часть проекта могла:
- хранить учётные записи пользователей
- разграничивать права доступа
- фиксировать игровые сессии
- сохранять поток игровых событий
- быстро получать текущее состояние игрока
- строить статистику по дням
- формировать таблицы лидеров

Такое ядро является универсальным и не зависит напрямую от жанра игры. Это позволяет использовать его как основу для дальнейшей жанровой адаптации под 2D-платформер.

---

## 2.5. Ограничения уникальности (UNIQUE)

В модели данных используются следующие обязательные ограничения уникальности:
- `users(email)`
- `roles(name)`
- `statistics_daily(user_id, day)`
- `leaderboard_scores(user_id, board_code, season)`

Эти ограничения необходимы для предотвращения дублирования критически важных данных.

---

## 2.6. Ограничения целостности (CHECK)

В ядре модели данных должны использоваться следующие базовые ограничения:
- `level >= 1`
- `season >= 1`
- `xp >= 0`
- `soft_currency >= 0`
- `hard_currency >= 0`
- `sessions_count >= 0`
- `events_count >= 0`
- `playtime_seconds >= 0`
- `wins >= 0`
- `losses >= 0`
- `score_sum >= 0`
- `score >= 0`

Эти ограничения позволяют защитить систему от некорректных значений.

---

## 2.7. Индексы

Для обеспечения производительности в базе данных должны использоваться индексы по типовым запросам:
- индекс по `users(email)` для авторизации
- индекс по `game_sessions(user_id, started_at)` для истории входов
- индекс по `game_events(user_id, created_at)` для просмотра последних событий пользователя
- индекс по `game_events(event_type, created_at)` для аналитики по типам событий
- индекс по `leaderboard_scores(board_code, season, score)` для формирования таблицы лидеров

---

## 2.8. Почему ядро не зависит от жанра

Несмотря на то, что проект представляет собой 2D-платформер, ядро базы данных остаётся универсальным. Это объясняется тем, что любая игра в серверной части всё равно опирается на одинаковые сущности:
- пользователь
- роли
- сессии
- события
- прогресс
- статистика
- рейтинги

Жанровая специфика будет добавляться отдельно в виде дополнительных таблиц, не нарушая ядро системы.
